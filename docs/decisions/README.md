# Architecture Decision Records (ADR)

이 디렉토리는 프로젝트의 **되돌리기 비싼 결정들**을 기록합니다.

> "코드는 *무엇을* 하는지 보여주지만, *왜 이렇게 했는지*는 설명하지 못합니다."

## 목적
- 6개월 뒤의 자신과 새 팀원이 "왜 이렇게 되어 있지?" 를 빠르게 이해
- 과거에 이미 기각된 대안을 **다시 논의하는 비용**을 줄이기
- 결정이 뒤집혔을 때 **어떤 맥락에서 바뀌었는지** 추적

## 형식
`NNNN-kebab-case-title.md` — NNNN은 4자리 순차 번호 (`0001`부터).
템플릿: [`0000-template.md`](./0000-template.md)

## 상태 (status)
| 값 | 의미 |
|---|---|
| `proposed` | 초안, 토론 중 |
| `accepted` | 승인되어 적용 중 |
| `deprecated` | 더 이상 유효하지 않음 (대체되지 않은 폐기) |
| `superseded-by-NNNN` | 이후 ADR로 대체됨 |

## 새 ADR 작성 순서
1. 가장 큰 번호 + 1 로 파일 생성
2. `0000-template.md` 내용 복사 후 작성
3. PR 본문에 **"Introduces ADR-NNNN"** 언급
4. 구현 PR은 ADR이 `accepted` 된 이후에 올린다 (큰 결정의 경우)

## 수정 규칙
- 본문을 **사후 재작성하지 않는다**. 결정 당시의 맥락이 역사다.
- 결정을 바꿀 때: **새 ADR**을 추가하고, 이전 ADR의 `status`를 `superseded-by-<new-id>`로 변경.
- 예외적 본문 수정(오탈자, 명백한 팩트 오류)은 **별도 커밋**으로 분리해 의도를 기록.

## 무엇을 기록할 것인가
자세한 범위는 [`CLAUDE.md` 의 "결정 로그" 섹션](../../CLAUDE.md)을 참고.

## 색인
현재 ADR 없음. 첫 ADR은 `0001-*.md`로 시작.
